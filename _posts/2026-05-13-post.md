강의를 참고하여 GAS를 이용한 포폴 프로젝트 구현하다가 이런 사용 패턴을 발견. 

참고로 PDBEffectActor는, 바닥에 놓인 물약이나 간단한 아이템 같은 것으로, 플레이어에게 Overlap되는 즉시 Attribute (체력 등)에 GameplayEffect를 적용하는 액터이다. 

```
void APDBEffectActor::ApplyEffectToTarget(AActor* TargetActor, TSubclassOf<UGameplayEffect> GameplayEffectClass)  
{  
    UAbilitySystemComponent* TargetASC = UAbilitySystemBlueprintLibrary::GetAbilitySystemComponent(TargetActor);  
    
	... 
      
    FGameplayEffectContextHandle EffectContextHandle = TargetASC->MakeEffectContext();  
    EffectContextHandle.AddSourceObject(this);  
      
    const FGameplayEffectSpecHandle EffectSpecHandle = TargetASC->MakeOutgoingSpec(GameplayEffectClass, ActorLevel, EffectContextHandle);  
    TargetASC->ApplyGameplayEffectSpecToSelf(*EffectSpecHandle.Data.Get());  
}
```

보자마자 이상함을 느꼈다. 

의미적으로만 봐도, GE를 적용할 거라면 관련된 Context나 Spec은 Source의 AbilitySystemComponent로부터 만들어지는 게 자연스럽지 않나? 

Source쪽에서 다 만든 다음에 그걸 Target한테 던지는게 맞지, Target ASC를 참조해서 만든 다음 SourceObject에 명시적으로 Source를 등록해서 그걸 스스로에게 적용하다니.

'Source는 꼴랑 물약인데 애초에 AbilitySystemComponent를 붙이기 애매하니까 이렇게 쓰는 건가' 정도의 추론은 가능해도, 무언가 찝찝하고 어색한 사용법이다.

---

엔진의 소스 코드를 살펴보자.

```
// AbilitySystemComponent.cpp:470
FGameplayEffectContextHandle UAbilitySystemComponent::MakeEffectContext() const
{
    FGameplayEffectContextHandle Context = FGameplayEffectContextHandle(
        UAbilitySystemGlobals::Get().AllocGameplayEffectContext());
          
	// By default use the owner and avatar as the instigator and causer
    if (ensureMsgf(AbilityActorInfo.IsValid(), ...))
    {
        Context.AddInstigator(AbilityActorInfo->OwnerActor.Get(), AbilityActorInfo->AvatarActor.Get());
    }
    return Context;
}
```

주석이 설명하듯이, TargetASC->MakeEffectContext()를 호출하는 순간 Context의 Instigator는 Target, 즉 우리 경우에서는 '물약에 닿은 캐릭터'다(플레이어의 경우 PlayerState가 되겠다). EffectCauser도 마찬가지(플레이어의 경우 Pawn이 되겠다). 

의문이 풀리지 않는다. 의미적으로도 어색했는데 실제로도 어색하다. '물약'이 '캐릭터'에게 회복 효과를 준 것이지 '캐릭터'가 '캐릭터'에게 회복 효과를 준 것이 아닌데. 

여러가지로 찾아보고 LLM에게도 질의하고 한 결과, 다음과 같은 논리로 이 패턴을 사용하는 것이라고 추론해볼 수 있었다. 

1. 원래대로라면 Source쪽에서 GE를 만들어 Target에 적용하는 것이 정석임.
2. 우리의 경우, Source는 '물약'이라고 할 수 있음.
3. 선택지는 3가지임.
	1. **'물약'에 해당하는 EffectActor에 ASC 컴포넌트 추가**
		1. 오버킬.
		2. 메모리/네트워크 낭비이다.
	2. **``FGameplayEffectContextHandle Ctx = FGameplayEffectContextHandle(UAbilitySystemGlobals::Get().AllocGameplayEffectContext()); Ctx.AddInstigator(this, this);`` 식으로 글로벌 Allocator 직접 호출 후 Instigator를 EffectActor로 세팅**
		1. 헬퍼 우회. 엔진 단위에서 설계한 추상화를 깨뜨리는 행위.
		2. 애초에 Instigator/EffectCauser는 GAS 컨벤션상 'ASC를 가진 지속 주체(Owner)'와 '물리적 원인(Avatar)'를 위한 슬롯. 실제로 데미지 어트리뷰션이나 팀 판정, AI Perception 등에서는 이 슬롯에 Pawn/Controller/PlayerState급 객체가 들어있다고 가정하므로 캐스팅 실패할 우려도 있음.
	3. **그냥 찝찝하지만 TargetASC->MakeEffectContext(); 하기**
4. 3번이 제일 Side Effect가 덜하므로 3번으로 간다.

여전히 찝찝하지만 결론이 이렇단다. 부작용도 덜하고 간편한데 의미적으로 조금 맞지 않아도 안 쓸 이유가 없다는 것. 최선은 아니지만 차악이라고 보여진다.

---

그렇다면 모든 경우에 이 패턴을 사용해도 될까? 

당연히 아니다. 

이번엔 TargetASC->MakeOutgoingSpec(...)의 내부를 따라가보자.

```
// AbilitySystemComponent.cpp:451
FGameplayEffectSpecHandle UAbilitySystemComponent::MakeOutgoingSpec(TSubclassOf<UGameplayEffect> GameplayEffectClass, float Level, FGameplayEffectContextHandle Context) const
{
    if (Context.IsValid() == false)
    {
        Context = MakeEffectContext();
    }
     if (GameplayEffectClass)
    {
        UGameplayEffect* GameplayEffect = GameplayEffectClass->GetDefaultObject<UGameplayEffect>();
        FGameplayEffectSpec* NewSpec = new FGameplayEffectSpec(GameplayEffect, Context, Level);
        return FGameplayEffectSpecHandle(NewSpec);
	}
    return FGameplayEffectSpecHandle(nullptr);
}
```

별 일이 없어 보이지만, FGameplayEffectSpec의 생성자가 내부의 Initialize()를
호출하고, 그 안에서 CaptureDataFromSource()가 불린다.

```
// GameplayEffect.cpp:1743
void FGameplayEffectSpec::CaptureDataFromSource(bool bSkipRecaptureSourceActorTags)
{
    // Capture source actor tags
    if (!bSkipRecaptureSourceActorTags)
    {
        RecaptureSourceActorTags();   // Context.Instigator 액터의 태그를   CapturedSourceTags로
    }

    // Capture source Attributes
    CapturedRelevantAttributes.CaptureAttributes(EffectContext.GetInstigatorAbilitySystemComponent(), EGameplayEffectAttributeCaptureSource::Source);
      ...
  }
```
  
엔진 내부 코드에 따르면 결국 Source Attribute는 Context의 Instigator에 연결된 ASC로부터 캡처된다. 그런데 우리는 바로 위에서 TargetASC->MakeEffectContext()로 Context의 Instigator를 Target으로 박아두었다.

종합하면 TargetASC->MakeOutgoingSpec(...)은 'Target의 태그와 어트리뷰트가 Source 슬롯에 들어가 있는 Spec' 을 만든다고 할 수 있는 것이다.
  
회복약의 경우 Magnitude가 +50 같은 단순 상수이며, Source Attribute도 Tag도 참조할 필요가 없기 때문에 무관했을 뿐, 다른 상황에서는 당연히 문제가 생길 수 있다.

Source의 AttackPower를 가져와 데미지를 계산한다든지, Source가 어떤 태그를 보유하고 있는 경우에만 특정 GE를 적용한다든지, 데미지를 ExecutionCalculation으로 계산한다든지, Magnitude를 시전자 레벨로 스케일링한다든지.... 하는 다양한, 그리고 드물지 않은 경우들에서 문제가 생길 수 있음을 우리는 쉽게 알 수 있다.

파이어볼에 맞았는데, 시전자가 아닌 맞은 사람의 공격력과 맞은 사람의 방어력을 가지고 계산해서 데미지를 적용하는 꼴이라고 보면 이해가 쉬울 것이다.

---

따라서 TargetASC를 가지고 GE의 Context나 Spec을 만들어 넘기는 패턴은 다음과 같은 경우에만 별 문제가 생기지 않을 거라고 짐작해볼 수 있다.

1. Source-side 정보를 전혀 참조하지 않는 GE: 상수 매그니튜드, Target 전용 캡처,
  Source 태그 요구사항 없음
2. self-applied 시나리오: Instigator=Target이 의미적으로 깨지지 않는 경우 

물약은 운 좋게 둘 다에 해당하는 케이스였을 뿐, 시스템 전반에 일반화할 패턴은 아니다.

결론은? 효과의 진짜 출처가 되는 캐릭터의 ASC가 Spec을 만든다는 원칙을 '가능한 한 끝까지 지킨다' 정도의 느슨한 감각을 가지고 있으면 좋겠다고 할 수 있다.

사실 '필드에 놓인 물약'이 좀 많이 특수한 경우일 뿐, 실제로 대다수의 경우에는 아래처럼 처리하면 될 것 같다는 생각이 든다. 

 - 데미지 어빌리티: UGameplayAbility가 자기 ActorInfo로
  MakeEffectContext()/MakeOutgoingSpec() 호출. 시전자 ASC가 Source.
 - 투사체: 발사 시점에 시전자 ASC가 만든 FGameplayEffectSpecHandle을 투사체가 들고
  다니다가 충돌 시 TargetASC->ApplyGameplayEffectSpecToSelf(*SpecHandle.Data.Get()).
- 장비 패시브 스탯: 장착 순간 캐릭터 ASC가 자기 자신에게 InfiniteDuration GE 적용,
  해제 시 핸들로 제거.

 즉, 비캐릭터 액터가 GE를 운반할 수는 있어도, GE를 만드는 ASC는 거의 항상 시전자라는 사실. EffectActor + TargetASC 패턴은 '운반과 생성을 같은 액터가 하면서, Source 캡처가 어차피 의미를 잃는 좁은 영역에서만 작동'하는 편의 우회로로 봐야 정확할 것이다!
